FizMaa Cashless — Complete Technical Architecture Documentation

Closed-loop NFC payment ecosystem for live events · Frontend · Backend · Security · CI/CD · Deployment

System Overview

FizMaa Cashless is a three-app NFC payment ecosystem for live events (festivals, college fests, expos). It replaces cash with MIFARE DESFire EV2 smart cards that store AES-128 encrypted balances directly on-chip, enabling fully offline payments at stalls while maintaining real-time visibility for organizers.

Component Type Primary Users Platform
FizMaa Companion Attendee App Event-goers React Native (iOS + Android)
FizMaa POS Booth Staff Terminal Vendor staff React Native (Android only, Phase 1)
FizMaa Admin Panel Web Dashboard Organizer, Supervisor, Accountant Next.js (Web)
FizMaa Backend Shared API All three apps FastAPI (Python)

1. System Architecture

All three apps share a single FastAPI backend. Role-based access is enforced entirely server-side via JWT claims — the app separation is a UX boundary, not a security boundary.

Data Flow Summary

  • Attendee taps card → NFC UID read → account linked in backend
  • Attendee pays via Razorpay inside Companion App → Celery job queued → Top-Up terminal writes new balance physically to NFC card
  • Vendor enters amount → POS App reads encrypted balance from card → deducts → writes back → transaction written to local SQLite queue
  • SQLite queue syncs to backend when connectivity restores → Celery reconciles transaction order using on-card counter
  • Admin Panel receives live transaction events via Socket.IO → real-time revenue and alert dashboards update

Physical Card — MIFARE DESFire EV2

The card is the source of truth for balance in offline mode. It stores the following fields in an AES-128 encrypted sector:

Field Purpose Size
encrypted_balance Rupee balance, AES-128 encrypted. Only POS terminals with the secret key can read or write. 4 bytes
last_txn_nonce Random value written after each payment. Rejects double-taps within 5 seconds. 8 bytes
last_txn_timestamp UTC timestamp of last transaction. Combined with nonce for double-tap window check. 8 bytes
txn_counter Incrementing integer. Server uses this to reconstruct chronological order from out-of-sequence syncs. 4 bytes
status_flag Written as "blocked" when card is reported lost. Accepted by offline terminals on next tap. 1 byte
Why MIFARE DESFire EV2 specifically?
Basic RFID cards only have a static UID — you cannot write a balance to them. Every stall would need internet to look up the balance, destroying the offline promise. DESFire EV2 has writable AES-128 encrypted memory sectors and a hardware backup mechanism: if power cuts out mid-write (phone battery dies, card yanked away mid-tap), the chip automatically rolls back to the last valid balance on the next read. Torn writes cannot corrupt data — this is a chip-level guarantee, not a software workaround.

One-line purpose: Let attendees load money, view spending history, chat with an AI assistant, and raise disputes — all from their phone.

Key Screens and Features

Card Linking (One-time)

When an attendee receives their NFC card at the entrance, they open the app and tap the card to their phone. The app reads the card UID using react-native-nfc-manager and sends it to the backend to link with their account. This is a one-time step — after this, the physical card and the app account are permanently paired.

iOS Note: The Companion App only reads the card UID once (for linking). This uses Core NFC Read, which Apple permits without special entitlements. The app does NOT write to cards, so no special Apple NFC entitlement is required. iOS support can launch alongside Android in Phase 1.

Balance Top-Up

  • Attendee selects an amount and payment method (UPI / Card / Net Banking) inside the app
  • Payment processed via Razorpay — FizMaa never handles raw card numbers
  • On Razorpay webhook confirmation, backend queues a Celery task
  • A Top-Up terminal writes the new balance physically to the NFC card
  • App reflects the new balance within 5 seconds via a server poll or Socket.IO push

Live Balance Display

The home screen shows the current balance in large type. This figure is fetched from the server, not read from the card (the card cannot be read unless physically tapped to the phone). The server balance and on-card balance stay in sync via the transaction counter reconciliation system.

Transaction History

Every top-up, payment, and refund is listed chronologically with stall name, amount, timestamp, and running balance. Cached locally in expo-sqlite for offline viewing. Attendees can raise a dispute on any specific transaction, which queues a refund request for supervisor review.

Spending Insights

Simple breakdown of what was spent at which stall category (Food / Beverages / Merch). Not analytics — just receipts made visual. Shows top stalls visited and a category breakdown chart.

AI Chatbot

An in-app Claude-powered assistant attendees can chat with. Handles:

  • Event FAQs (where is the medical tent, what time does the main stage open)
  • Spending queries ("how much have I spent on food today?")
  • Balance and top-up guidance ("how do I add more money?")
  • Dispute assistance ("I was charged twice, what do I do?")

The chatbot calls a backend proxy endpoint (POST /ai/chat) which adds event context (current balance, recent transactions, event info) to the system prompt before forwarding to the Claude API. The attendee's financial data never leaves the backend — only a contextual summary is included in the prompt.

3. App 2 — FizMaa POS (Booth Staff Terminal)

One-line purpose: The phone IS the POS machine. Staff enter an amount, the attendee taps their card, and the payment is deducted directly from the card — with or without internet.

3.1 Frontend Architecture

The POS App is the most technically critical app in the system. It handles all actual payment writes to the NFC card.

Tech Stack

Language TypeScript Financial logic cannot have implicit type coercions. Strict mode enforced.

Encryption react-native-aes-crypto AES-128 encryption and decryption of the balance sector on the card. Secret key stored in Expo Secure Store.

Local DB expo-sqlite Stores offline transaction queue. Persists across app restarts and device reboots.

Payment Flow — Step by Step

  1. Staff opens the Keypad screen. Enters rupee amount using the large tap-friendly keypad.
  2. Staff taps "Charge". App transitions to the Visual Handshake Screen.
  3. Attendee views the screen showing: Current Balance, Deduction Amount, New Balance. They can walk away if the amount is wrong.
  4. Attendee taps their card to the phone's NFC reader.
  5. App reads AES-128 encrypted balance from the DESFire EV2 card.
  6. App validates cryptographic signature. Rejects forged cards immediately.
  7. App checks balance is sufficient. If not, shows Red Declined screen with reason and shortfall amount.
  8. App checks last_txn_nonce. Rejects double-taps within 5 seconds.
  9. App writes new encrypted balance + new nonce + timestamp back to card.
  10. App locks NFC for 3 seconds (Layer 1 double-tap protection).
  11. Transaction written to local SQLite queue immediately.
  12. Green screen with success sound shown. Card UID, amount, and new balance displayed.
  13. Background sync job uploads the queued transaction to the server when internet is available.

Visual Handshake Screen — Anti-Fraud Design

This is the most important UX decision in the entire system. When staff hits "Charge", the app does NOT immediately activate NFC. It first shows a full-screen confirmation with the exact balance, deduction, and resulting new balance before any NFC action is taken.

The attendee physically sees this screen before they tap. The vendor cannot skip or dismiss it. This prevents silent overcharging entirely — the most common fraud vector in closed-loop event payment systems.

Three-Layer Double-Tap Protection

Layer Mechanism Window Where Enforced
Layer 1 App NFC lock after successful write 3 seconds POS App (client-side)
Layer 2 On-card nonce check — last_txn_nonce must differ 5 seconds POS App reads card nonce before writing
Layer 3 Server idempotency key on sync endpoint Permanent FastAPI backend rejects duplicate txn_id

Offline Queue Architecture

  • Every transaction is written to local SQLite FIRST, always. The payment flow is identical online or offline.
  • expo-background-fetch attempts sync when internet becomes available, even when the app is backgrounded.
  • The on-card txn_counter increments with every payment. The server uses this counter to reconstruct the correct chronological order even when transactions from multiple stalls arrive out of sequence.
  • End-of-day "Force Sync Now" button pushes all queued transactions regardless of background sync delays.

4. App 3 — FizMaa Admin Panel (Web Dashboard)

One-line purpose: Live visibility into every rupee moving through the event, with tools to handle disputes, freeze lost cards, flag fraud, and run end-of-day settlement.

4.1 Frontend Architecture

All three roles — Admin/Organizer, Supervisor, and Accountant — use the same web application but see different views and have different permissions. Role enforcement happens at the API level via JWT claims. The frontend renders only the views that the API responds to for that role.

Charts Recharts Live revenue graphs, transactions-per-minute sparklines, category pie charts.

Role-Based Views

Admin / Organizer

  • Live Event Dashboard — real-time counters via Socket.IO: total revenue, active cards, top-up vs spending volume, transactions per minute, stall leaderboard
  • Stall Management — create stalls, assign staff, set category tags (Food / Beverage / Merch)
  • Event Configuration — max top-up per card, refund window duration, velocity alert thresholds, terminal assignments
  • Velocity Anomaly Alerts — when a vendor's offline batch sync shows average transaction value more than 3× their online baseline, the system auto-flags the stall, freezes their payout, and surfaces an alert for manual review
  • Card Management — view all issued cards, current server balance, status (active / blocked / expired). Block or freeze any card UID instantly. This is the admin-side of the lost card freeze flow — when an attendee reports a lost card through the Companion App, the freeze request surfaces here for the admin to confirm and action.
  • End-of-Day Settlement — per-vendor payout calculation, refund deduction, payout trigger, PDF and Excel report generation

Supervisor

  • Refund Approval Queue — review each refund request raised by attendees or booth staff, approve or reject. On approval, a terminal writes the refunded balance back to the physical card.
  • Lost Card Replacement — scan a new blank card, enter the lost card UID (from the attendee's account), transfer server balance, block the old UID. Done in under 2 minutes.
  • Stall Audit — real-time full transaction list for any stall. Used during manual audits of flagged vendors.
  • Dispute Log — full history of every flagged transaction with resolution status.

Accountant

  • Read-only access to settlement reports and transaction exports.
  • Download per-event reports in PDF or Excel. Filter by stall, category, time window, or transaction type.
  • No write access — cannot approve refunds, block cards, or modify event configuration.

5. Backend Architecture

One FastAPI backend serves all three apps. It is the only system with write access to the PostgreSQL ledger. Security is enforced entirely at the API level through JWT role claims.

5.1 Tech Stack

Layer Tool Reason
Framework FastAPI (Python) Async by default. Handles hundreds of concurrent sync requests from stalls reconnecting after network outages.
Database PostgreSQL ACID-compliant. Financial data requires proper transactions, rollbacks, and complete audit logs. No NoSQL.
Cache Redis Caches card balances (TTL 30s). Acts as pub/sub broker for Socket.IO live dashboard events.
Background Jobs Celery (Redis broker) Handles offline sync batches, reconciliation, report generation, settlement payouts — all async.
Scheduled Jobs Celery Beat Auto-settle after event end, auto-archive old event data, periodic reconciliation health checks.
Auth JWT + Refresh Tokens Role encoded in token claims: admin, supervisor, booth_staff, accountant, attendee. Validated on every request.
Realtime Socket.IO Pushes live transaction events to the Admin Panel dashboard. Redis pub/sub allows horizontal scaling.
ORM SQLAlchemy + Alembic Clean async ORM models. Alembic handles version-controlled database migrations.
File Storage AWS S3 Settlement report PDFs, Excel exports. Pre-signed URLs for secure time-limited downloads.
AI Proxy Anthropic Claude API Backend proxy for attendee chatbot. Adds event context to system prompt. Attendee financial data never sent raw to Claude.

5.2 Core Endpoints — Domain Summary

Authentication /auth/*

  • POST /auth/register — attendee self-registration
  • POST /auth/login — any role. Returns JWT + refresh token. Role claim set from user record in DB.
  • POST /auth/refresh — exchange refresh token for new access token. Old token invalidated.

Cards /cards/*

  • POST /cards/link — attendee links their NFC card UID to their account
  • GET /cards/{uid}/balance — returns server balance (cached in Redis, TTL 30s)
  • GET /cards/{uid}/status — returns active / blocked / expired (used by POS before writes)
  • POST /cards/{uid}/block — admin or supervisor blocks/freezes a card. Instant propagation to online terminals.
  • POST /cards/replace — supervisor transfers balance from lost card UID to new card UID

Transactions /transactions/* and /sync/*

  • GET /transactions — paginated history, filterable by card, stall, time, type
  • POST /sync/batch — POS App posts offline transaction batches. Idempotent: duplicate txn_id is silently ignored.
  • POST /sync/force — force-flush entire SQLite queue at event close
  • Celery task on batch receipt: reconcile transaction order using on-card txn_counter, update PostgreSQL ledger, publish Socket.IO event

Top-Ups /topups/*

  • POST /topups/initiate — creates Razorpay order, returns order_id
  • POST /topups/webhook — Razorpay calls this with HMAC-signed payload. Validates signature, queues Celery task to write balance to card at next Top-Up terminal.

Refunds /refunds/*

  • POST /refunds/request — attendee raises a dispute on a specific transaction
  • POST /refunds/flag — booth staff flags a transaction for refund
  • POST /refunds/{id}/approve — supervisor approves. Queues Celery task to write refunded balance back to card.
  • POST /refunds/{id}/reject — supervisor rejects with reason.

AI Chatbot /ai/*

  • POST /ai/chat — receives attendee message, constructs a system prompt with event context (current balance, recent 10 transactions, event name, stall map summary), forwards to Claude API, returns response. Conversation history maintained per session in Redis with TTL.

Settlement /settlement/*

  • POST /settlement/generate — admin triggers. Celery calculates per-stall net (gross minus approved refunds), generates PDF and Excel, uploads to S3, emails download link.
  • POST /settlement/stalls/{id}/payout — admin approves or rejects individual stall payout. Frozen stalls (anomaly flag) require explicit approval.

Realtime /ws/*

  • Socket.IO endpoint for Admin Panel. Receives events: txn_created, alert_raised, card_blocked, sync_received

5.3 Idempotency and Reconciliation

Every transaction the POS App generates is assigned a UUID (txn_id) at the moment of writing to SQLite on the phone. When the batch is synced to the server, the endpoint checks: "Does this txn_id already exist in the ledger?" If yes, it returns 200 OK and discards the duplicate silently. This means POS Apps can retry failed sync requests indefinitely without creating phantom charges.

Transaction order reconciliation uses the on-card txn_counter. Even when three stalls sync simultaneously with interleaved timestamps, the server reconstructs the correct sequence of debits and credits for any given card by sorting by txn_counter.

6. Security Architecture

6.1 Firewall — Is it Needed?

Yes. A firewall is required, but it operates at the infrastructure level (cloud VPC security group rules), not as a separate physical appliance. The firewall's job is to ensure the database and cache are never reachable from the public internet.

What to Open and What to Close

Resource Port Public? Who Can Access Why
FastAPI (HTTPS) 443 Yes All apps, Razorpay webhook All app communication
FastAPI (HTTP) 80 Yes (redirect only) Redirect to 443 HTTPS enforcement
PostgreSQL 5432 NO FastAPI container only (VPC internal) Prevents direct DB brute-force
Redis 6379 NO FastAPI + Celery workers (VPC internal) Cache and broker not publicly accessible
Celery workers Internal NO Celery Beat + FastAPI (VPC internal) Job processing only
S3 / File storage 443 (HTTPS) Via pre-signed URLs only Authenticated users with time-limited tokens Reports download
Core firewall rule: PostgreSQL (port 5432) and Redis (port 6379) must be completely closed to the public internet. Only the FastAPI container, whitelisted by VPC security group rules, may communicate with them.

6.2 Mobile App Security

Attendee App (FizMaa Companion)

  • All communication over HTTPS (TLS 1.3 minimum)
  • JWT stored in Expo Secure Store (Keychain on iOS, Keystore on Android) — never in AsyncStorage
  • NFC only reads card UID — no write access, no balance data exposed on the attendee's phone
  • AI chat messages proxied through backend — Claude API key never in the mobile bundle

POS App (FizMaa POS) — Higher Security Requirements

  • All communication over HTTPS (TLS 1.3 minimum)
  • SSL Pinning implemented via a custom OkHttp interceptor: the app's expected server certificate hash is bundled at build time. If a man-in-the-middle proxy (Wireshark, Charles Proxy, Burp Suite) intercepts the connection, the certificate hash won't match and the connection is killed before any data is transmitted.
  • AES-128 secret key (for encrypting/decrypting card balance sector) stored in Expo Secure Store — never in JS bundle, never in source code, injected at EAS Build time via EAS Secrets.
  • JWT scoped to booth_staff role — cannot call admin, supervisor, or attendee-exclusive endpoints.
  • Device binding (Phase 2): each device registered by IMEI/Android ID. A POS JWT will only work from the registered device.

6.3 The Four Security Problems — Solved

Problem Attack Scenario Solution
Lost card Finder taps lost card at a stall and spends the balance Attendee reports via Companion App → admin/supervisor freezes card UID in Admin Panel → online terminals block within seconds → Supervisor Booth replaces card with balance transferred
Torn write Phone battery dies mid-NFC-write, card data corrupted DESFire EV2 hardware backup sector auto-rollback. Physically impossible to corrupt — chip-level guarantee.
Out-of-order sync Top-up not yet on server when payment arrives; server balance looks negative On-card txn_counter. Server holds reconciliation until all transactions for a card are received in order via Celery.
Vendor overcharges offline Stall goes to airplane mode, charges ₹1,500 instead of ₹150 Visual Handshake Screen (attendee sees amount before tapping) + velocity anomaly detection freezes payout on sync.
Double-tap charge Card read twice within 1 second, balance deducted twice 3-layer defence: app NFC lock (3s) + on-card nonce check (5s) + server idempotency key (permanent).

8. CI/CD Pipeline

The pipeline runs on GitHub Actions. Every push to any branch triggers CI. Deployment (CD) only triggers on merges to develop (staging) or main (production).

8.1 Backend CI/CD

CI Phase — triggered on every PR to develop or main

  1. Checkout code and set up Python 3.12 with pip cache
  2. Install dependencies from requirements.txt
  3. Run ruff linter — enforces code style and import ordering
  4. Run mypy type checker — ensures Pydantic model type safety
  5. Spin up temporary PostgreSQL and Redis instances via GitHub Actions service containers
  6. Run Alembic migrations against the test database
  7. Execute pytest suite — covers: idempotency tests, velocity anomaly detection, refund approval flow, JWT role enforcement, AI proxy prompt construction
  8. Check test coverage — must be above 80% or CI fails

CD Phase — triggered on merge to main

  1. Build Docker image from backend/Dockerfile
  2. Tag image with git SHA: fizmaa-backend:abc1234
  3. Push image to AWS ECR (Elastic Container Registry)
  4. Run Alembic database migrations against production PostgreSQL (inside the pipeline, before traffic switches)
  5. Deploy new task definition to AWS ECS. ECS performs a blue-green swap: new container health-checked, traffic shifted, old container terminated.
  6. Post-deploy smoke test: call GET /health on the new deployment. If it fails, ECS automatically rolls back to the previous task definition.
  7. Notify Slack #deployments with deployment summary
Zero-downtime guarantee: The ECS blue-green deployment means no in-flight sync requests from field POS devices are dropped during a deployment. The old container stays alive until all its connections drain naturally.

8.2 Mobile App CI/CD (Companion + POS)

CI Phase — triggered on every PR

  1. Checkout code, set up Node.js 20 with npm cache
  2. Install dependencies from package.json
  3. Run ESLint and TypeScript compiler (tsc --noEmit) — enforces strict type checking on all financial logic
  4. Run Jest unit tests — covers: SQLite queue initialization, AES encryption/decryption round-trip, Zustand store state transitions, Razorpay mock, AI format
  5. Run Expo Doctor — validates that all native module configurations are compatible with the target Expo SDK version

CD Phase — triggered on merge to main or release/*

Distribution Submitted to Google Play Internal Testing and Apple TestFlight. Promoted to public after QA sign-off. NOT submitted to Play Store. Distributed via authenticated staff portal or MDM.

OTA Updates — When to Use

  • Minor bug fixes and UI tweaks that do not touch native modules → trigger an Expo OTA update. The JavaScript bundle on the device updates silently in the background on next app open.
  • Changes to NFC drivers, camera, Bluetooth, or any native module → requires a full EAS Build and new app binary. OTA cannot update native code.
  • POS App: OTA updates are preferred during a live event — vendors do not need to re-download the app.

9.2 Backend Deployment — Step by Step

  1. Developer merges PR to main on GitHub.
  2. GitHub Actions CI/CD pipeline triggers automatically.
  3. Docker image built from backend/Dockerfile. Image tagged with git SHA.
  4. Image pushed to AWS ECR.
  5. Pipeline runs Alembic migrations against production RDS. If migration fails, pipeline halts — no deployment proceeds.
  6. New ECS Task Definition created pointing to the new image tag.
  7. ECS deploys the new task alongside the old one (blue-green). Health check: GET /health must return 200.
  8. On health check pass, ECS routes all traffic to the new task. Old task drains connections and terminates.
  9. If health check fails within 3 minutes, ECS automatically rolls back to the previous task definition. Zero manual intervention required.
  10. Slack notification sent to #deployments with git SHA, migration summary, and deploy time.

11. Timeline and Milestones

Here is heavily structured, four-week sprint to get FizMaa Cashless production-ready by July 31st.

Week 1: The Core Brain (July 1 – July 7)

Goal: A fully functional FastAPI backend, database schema, and the Celery/Redis idempotency engine. Without this, the apps have nothing to talk to.

Milestone Key Tasks Definition of Done
1. DB & Auth Setup PostgreSQL (Neon) and SQLAlchemy models. Implement JWT auth middleware. User can register, login, and get a role-based JWT (attendee, booth_staff, admin).
2. Core Endpoints Build CRUD for Stalls, Cards, and Users. API can assign a UID to a user and fetch their server balance.
3. Offline Sync Engine Build POST /sync/batch. Setup Redis idempotency lock. API successfully accepts an array of offline transactions and ignores duplicate txn_ids.
4. Celery Workers Configure Celery to handle the sync queue in the background. Out-of-order nonces are correctly stitched together in PostgreSQL without blocking the main API thread.

Week 2: The Hardware Spike (July 8 – July 14)

Goal: The Android POS App. This is the highest technical risk of the project. If you cannot securely read/write to the DESFire EV2 card offline, the ecosystem fails.

Milestone Key Tasks Definition of Done
1. React Native Setup Initialize the POS app, configure EAS Build, set up expo-sqlite and Zustand. App builds successfully via EAS and runs on a physical Android device.
2. NFC Read/Write Integrate react-native-nfc-manager and react-native-aes-crypto. App can read the encrypted balance, write a new balance, and increment the card's nonce.
3. The Visual Handshake Build the POS UI: Keypad → Handshake Screen → NFC Tap → Success/Fail. Vendor can type ₹150, attendee sees it, taps, and balance deducts offline.
4. Offline Queue & Sync Hook SQLite queue to the FastAPI sync endpoint. App can process 5 payments in Airplane Mode, and push them to the server instantly upon reconnecting to WiFi.

Week 3: Attendee App & Admin Dashboard (July 15 – July 21)

Goal: The consumer face of the app and the live dashboard for event organizers.

Milestone Key Tasks Definition of Done
1. Next.js Admin Panel Setup Next.js App Router, Tailwind, and TanStack Tables. Admin can log in and view a paginated, filterable table of all transactions and issued cards.
2. Live Socket Feed Hook up Socket.IO between FastAPI and the Next.js dashboard. When a terminal syncs a transaction, the Admin Panel revenue graph updates instantly without refreshing.
3. Companion App UI Initialize the Attendee React Native app. Build login, balance view, and transaction history. Attendee can log in, tap their card to link the UID, and see their live server balance.
4. Razorpay Integration Wire Razorpay SDK in the Companion App to the backend webhook. Attendee can pay ₹500 online, and the database server_balance updates securely.

Week 4: Deployment, Security & Simulation (July 22 – July 31)

Goal: Move from localhost to production, lock down the security, and break the system intentionally to ensure it recovers.

Milestone Key Tasks Definition of Done
1. Cloud Deployment Dockerize FastAPI. Deploy to AWS ECS or Render. Setup Cloudflare for Admin panel. Backend is live on a public HTTPS domain. Database ports are firewalled from the public.
2. CI/CD & OTA Setup GitHub Actions for backend linting/testing. Configure Expo OTA updates. Pushing to main auto-deploys the backend and pushes JS bundle updates to the mobile apps.
3. Card Freeze Flow Build the "Kill Switch" flow across all three apps. Attendee clicks "Freeze", Admin approves, and POS terminal immediately rejects the card.
4. The "Chaos" Test Buy blank cards. Simulate a live event in your room with real network drops. Pull the card away mid-write (test anti-tear). Turn off the router mid-sync (test idempotency). The ledger must remain perfectly accurate.

Glossary

Term Definition
DESFire EV2 MIFARE DESFire EV2 — a smart card chip by NXP that supports AES-128 encrypted writable memory sectors and hardware backup rollback on torn writes.
EAS Build Expo Application Services Build — Expo's cloud build service. Required for any React Native app that uses native modules (NFC, camera, Bluetooth).
OTA Update Over-The-Air update — a mechanism to update only the JavaScript bundle of an Expo app without a full app store release. Cannot update native code.
Idempotency A property of an API endpoint: calling it multiple times with the same input produces the same result. Critical for sync retry safety.
Celery A distributed task queue for Python. Used to run background jobs asynchronously (sync processing, report generation, settlement).
Alembic A database migration tool for SQLAlchemy. Tracks schema changes as version-controlled migration scripts.
txn_counter An incrementing integer stored on the NFC card. Used by the server to reconstruct chronological transaction order when out-of-sequence syncs arrive.
Velocity anomaly When a vendor's average transaction value in an offline batch exceeds 3× their historical online average — a potential fraud signal.
VPC Virtual Private Cloud — a logically isolated network on AWS. All backend services (PostgreSQL, Redis, Celery) run inside the VPC, invisible to the public internet.
SSL Pinning A mobile security technique where the app's expected server certificate hash is hardcoded at build time. Prevents man-in-the-middle interception.
Blue-green deploy A deployment strategy where a new version runs alongside the old one. Traffic switches only after the new version passes a health check.
Visual Handshake The full-screen confirmation screen shown on the POS App before NFC is activated. Attendee sees the exact deduction amount before tapping their card.
Card freeze Admin/supervisor action that marks a card UID as blocked on the server. Online terminals refuse the card immediately; offline terminals pick up the block flag on the next tap.
