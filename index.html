FizMaa Cashless — Complete Technical Architecture Documentation

Closed-loop NFC payment ecosystem for live events · Frontend · Backend · Security · CI/CD · Deployment

System Overview

FizMaa Cashless is a three-app NFC payment ecosystem for live events (festivals, college fests, expos). It replaces cash with MIFARE Ultralight and MIFARE Classic 1K smart cards that store HMAC-SHA256 signed balances directly on-chip, enabling fully offline payments at stalls while maintaining real-time visibility for organizers.

Component Type Primary Users Platform
FizMaa Companion Attendee App Event-goers React Native (iOS + Android)
FizMaa POS Booth Staff Terminal Vendor staff React Native (iOS + Android)
FizMaa Admin Panel Web Dashboard Organizer, Supervisor, Accountant Next.js (Web)
FizMaa Backend Shared API All three apps FastAPI (Python)

1. System Architecture

All three apps share a single FastAPI backend. Role-based access is enforced entirely server-side via JWT claims — the app separation is a UX boundary, not a security boundary.

Data Flow Summary

  • Attendee taps card → NFC UID read → account linked in backend
  • Attendee pays via Razorpay inside Companion App → Celery job queued → Top-Up terminal writes new balance physically to NFC card
  • Vendor enters amount → POS App reads encrypted balance from card → deducts → writes back → transaction written to local SQLite queue
  • SQLite queue syncs to backend when connectivity restores → Celery reconciles transaction order using on-card counter
  • Admin Panel receives live transaction events via Socket.IO → real-time revenue and alert dashboards update

Physical Card — MIFARE Ultralight / MIFARE Classic 1K

The card is the source of truth for balance in offline mode. It stores the following fields (protected by a software HMAC-SHA256 signature):

Field Purpose Size
balance_primary & balance_shadow Rupee balance (primary and shadow copy for torn-write recovery). Only POS terminals with the secret signing key can forge valid balances. 4 bytes each
last_txn_nonce Random value written after each payment. Rejects double-taps within 5 seconds. 8 bytes
last_txn_timestamp UTC timestamp of last transaction. Combined with nonce for double-tap window check. 8 bytes
txn_counter Incrementing integer. Server uses this to reconstruct chronological order from out-of-sequence syncs. 4 bytes
status_flag Written as "blocked" when card is reported lost. Accepted by offline terminals on next tap. 1 byte
hmac_signature HMAC-SHA256 signature validating the data (prevents tampering). 19-32 bytes
Why not basic RFID?
Basic RFID cards only have a static UID — you cannot write a balance to them. MIFARE Ultralight and Classic 1K have writable memory. Although they lack hardware AES-128 and torn-write rollback, FizMaa compensates with software-layer HMAC signing and a two-field write-verify protocol (primary and shadow balances) to prevent corruption and tampering.

Card Provisioning & Top-Up Process

Because the system uses MIFARE Classic 1K / Ultralight + HMAC Signatures, the provisioning process at the gate is the most important security moment of the entire event. This is where the digital ticket transforms into a physical, cryptographically signed wallet.

Scenario A: The Pre-Bought Ticket (Gold, Platinum, VIP)

These attendees bought their tickets online days or weeks in advance. Their ticket package might include entry plus a pre-loaded balance (e.g., a "Gold Ticket" includes ₹2,000 to spend at the bar).

  1. Verify the Online Ticket: Action taken by Gate Staff. The attendee walks up to the registration desk and shows the ticket QR code on their phone. The staff uses the FizMaa POS App (set to Registration Mode) to scan the QR code. The app instantly checks the FastAPI backend to ensure the ticket is valid and hasn't been used yet.
  2. The Physical Link: Connecting digital to physical. The POS App sees that this QR code is a "Gold Ticket" that comes with a ₹2,000 pre-loaded balance. The app prompts the staff member to tap a blank card. The staff picks up an unregistered MIFARE Classic 1K or Ultralight card and holds it to the back of their phone.
  3. The Cryptographic Write: Securing the offline wallet. In a fraction of a second, the POS App writes Balance: ₹2,000 and Counter: 1 to the card. It calculates the HMAC Signature combining the card's permanent UID, the balance, and the secret app key, and writes that signature to the card so vendors can verify it offline later.
  4. Backend Sync: Updating the cloud ledger. The POS App tells the FastAPI server: "Ticket QR 9876 is now consumed. It is permanently linked to Card UID 1234, and the starting balance is ₹2,000." The attendee walks into the festival and can start tapping at food stalls immediately.

Scenario B: The On-the-Spot Walk-In

These attendees show up at the venue with no prior registration. They need to buy a ticket, get a card, and load it with money all at the exact same time.

  1. Take Payment: Physical Cash or UPI. The attendee walks up to the Top-Up counter and says, "I need entry, and I want to load ₹1,000." They scan a static UPI QR code taped to the desk or hand physical cash to the staff member.
  2. Configure the Top-Up: Action taken by Gate Staff. Once the staff verifies the UPI ping or puts the cash in the drawer, they open the FizMaa POS App (set to Top-Up Mode) and type "₹1,000" into the keypad.
  3. The Cryptographic Write: Creating the wallet. The staff picks up a blank card and taps it to their phone. Just like the pre-bought flow, the app writes the ₹1,000 balance and generates the strict HMAC security signature to lock the data.
  4. Anonymous Profile Creation: Optional later link. The POS App pings the FastAPI backend to create a brand new, anonymous user profile in the database, linked to this specific Card UID with a server balance of ₹1,000. (Optional Upgrade: If the attendee wants to check their balance later, they can download the Companion App, tap the card, and the system will link that anonymous profile to their phone number.)

⚠️ Crucial Architecture Constraint: Handling "Ticket Types"

Because you are using MIFARE Ultralight cards, you only have 48 bytes of usable space. This creates a hard limitation: You cannot store the "Ticket Type" (Gold, Platinum, VIP) directly on the physical card. The space is entirely consumed by the Rupee balance and the HMAC security signature.

If a vendor's POS terminal is completely offline, it will have no way of knowing if the card tapping it belongs to a "Platinum" attendee or a general admission attendee.

How to solve this (Handle tier privileges physically or hybridly):

  • Physical Visuals: The easiest operational fix is to use different colored physical cards or wristbands. Give Gold buyers a black Ultralight wristband and General Walk-ins a white one. The security guards check the color for VIP zone access, while the NFC chip strictly handles the money.
  • Online-Only Checks: If a specific VIP bar offers "20% off for Platinum members," that specific POS terminal must have an active internet connection. When the card taps, the terminal reads the UID, quickly asks the FastAPI server "What tier is this UID?", and applies the discount before writing the new balance.

Key Screens and Features

Card Linking (One-time)

When an attendee receives their NFC card at the entrance, they open the app and tap the card to their phone. The app reads the card UID using react-native-nfc-manager and sends it to the backend to link with their account. This is a one-time step — after this, the physical card and the app account are permanently paired.

iOS Note: The Companion App only reads the card UID once (for linking). This uses Core NFC Read, which Apple permits without special entitlements. The app does NOT write to cards, so no special Apple NFC entitlement is required. iOS support can launch alongside Android in Phase 1.

Balance Top-Up

  • Attendee selects an amount and payment method (UPI / Card / Net Banking) inside the app
  • Payment processed via Razorpay — FizMaa never handles raw card numbers
  • On Razorpay webhook confirmation, backend queues a Celery task
  • A Top-Up terminal writes the new balance physically to the NFC card
  • App reflects the new balance within 5 seconds via a server poll or Socket.IO push

Live Balance Display

The home screen shows the current balance in large type. This figure is fetched from the server, not read from the card (the card cannot be read unless physically tapped to the phone). The server balance and on-card balance stay in sync via the transaction counter reconciliation system.

Transaction History

Every top-up, payment, and refund is listed chronologically with stall name, amount, timestamp, and running balance. Cached locally in expo-sqlite for offline viewing. Attendees can raise a dispute on any specific transaction, which queues a refund request for supervisor review.

Spending Insights

Simple breakdown of what was spent at which stall category (Food / Beverages / Merch). Not analytics — just receipts made visual. Shows top stalls visited and a category breakdown chart.

AI Chatbot

An in-app Claude-powered assistant attendees can chat with. Handles:

  • Event FAQs (where is the medical tent, what time does the main stage open)
  • Spending queries ("how much have I spent on food today?")
  • Balance and top-up guidance ("how do I add more money?")
  • Dispute assistance ("I was charged twice, what do I do?")

The chatbot calls a backend proxy endpoint (POST /ai/chat) which adds event context (current balance, recent transactions, event info) to the system prompt before forwarding to the Claude API. The attendee's financial data never leaves the backend — only a contextual summary is included in the prompt.

3. App 2 — FizMaa POS (Booth Staff Terminal)

One-line purpose: The phone IS the POS machine. Staff enter an amount, the attendee taps their card, and the payment is deducted directly from the card — with or without internet.

3.1 Frontend Architecture

The POS App is the most technically critical app in the system. It handles all actual payment writes to the NFC card.

Tech Stack

Layer Tool Reason
Framework React Native (Expo + EAS Build) EAS Build is non-negotiable for NFC write access on Android. Standard Expo Go lacks NFC write APIs.
Language TypeScript Financial logic cannot have implicit type coercions. Strict mode enforced.
NFC react-native-nfc-manager Supports Ultralight and Classic 1K read/write. Sector authentication for Classic 1K via Key A. Reads balance, validates HMAC signature, writes new balance + nonce.
HMAC / Crypto react-native-crypto HMAC-SHA256 computation (via node-forge polyfill) for signature generation and verification. Secret key stored in Expo Secure Store.
State Zustand Tracks isNfcLocked boolean (the 3-second post-payment cooldown gate) and offline queue length.
Local DB expo-sqlite Stores offline transaction queue. Persists across app restarts and device reboots.
Background Sync expo-background-fetch Attempts sync to backend when app is backgrounded and internet becomes available.
Navigation Expo Router Simple flow: Keypad → Handshake → Result → Queue.
Auth JWT (role: booth_staff) Server enforces: booth_staff tokens cannot call supervisor or admin endpoints.
Platform Android + iOS (Phase 1) Native CoreNFC supports MIFARE read/write out of the box. No Apple entitlement required.

Native iOS Support Unlocked

By shifting to MIFARE Classic/Ultralight cards, the system bypasses previous iOS restrictions. Apple's CoreNFC framework fully supports reading and writing MIFARE tags out of the box via NFCMiFareTag. This eliminates the need for the 4-6 week Apple Entitlement review, allowing the POS App to be deployed on both Android and iOS devices concurrently from Day 1.

Payment Flow — Step by Step

  1. Staff opens the Keypad screen. Enters rupee amount using the large tap-friendly keypad.
  2. Staff taps "Charge". App transitions to the Visual Handshake Screen.
  3. Attendee views the screen showing: Current Balance, Deduction Amount, New Balance. They can walk away if the amount is wrong.
  4. Attendee taps their card to the phone's NFC reader.
  5. App reads balance and HMAC signature from the card.
  6. App validates cryptographic signature. Rejects forged cards immediately.
  7. App checks balance is sufficient. If not, shows Red Declined screen with reason and shortfall amount.
  8. App checks last_txn_nonce. Rejects double-taps within 5 seconds.
  9. App writes new encrypted balance + new nonce + timestamp back to card.
  10. App locks NFC for 3 seconds (Layer 1 double-tap protection).
  11. Transaction written to local SQLite queue immediately.
  12. Green screen with success sound shown. Card UID, amount, and new balance displayed.
  13. Background sync job uploads the queued transaction to the server when internet is available.

Visual Handshake Screen — Anti-Fraud Design

This is the most important UX decision in the entire system. When staff hits "Charge", the app does NOT immediately activate NFC. It first shows a full-screen confirmation with the exact balance, deduction, and resulting new balance before any NFC action is taken.

The attendee physically sees this screen before they tap. The vendor cannot skip or dismiss it. This prevents silent overcharging entirely — the most common fraud vector in closed-loop event payment systems.

Three-Layer Double-Tap Protection

Layer Mechanism Window Where Enforced
Layer 1 App NFC lock after successful write 3 seconds POS App (client-side)
Layer 2 On-card nonce check — last_txn_nonce must differ 5 seconds POS App reads card nonce before writing
Layer 3 Server idempotency key on sync endpoint Permanent FastAPI backend rejects duplicate txn_id

Offline Queue Architecture

  • Every transaction is written to local SQLite FIRST, always. The payment flow is identical online or offline.
  • expo-background-fetch attempts sync when internet becomes available, even when the app is backgrounded.
  • The on-card txn_counter increments with every payment. The server uses this counter to reconstruct the correct chronological order even when transactions from multiple stalls arrive out of sequence.
  • End-of-day "Force Sync Now" button pushes all queued transactions regardless of background sync delays.

One-line purpose: Live visibility into every rupee moving through the event, with tools to handle disputes, freeze lost cards, flag fraud, and run end-of-day settlement.

All three roles — Admin/Organizer, Supervisor, and Accountant — use the same web application but see different views and have different permissions. Role enforcement happens at the API level via JWT claims. The frontend renders only the views that the API responds to for that role.

Role-Based Views

Admin / Organizer

  • Live Event Dashboard — real-time counters via Socket.IO: total revenue, active cards, top-up vs spending volume, transactions per minute, stall leaderboard
  • Stall Management — create stalls, assign staff, set category tags (Food / Beverage / Merch)
  • Event Configuration — max top-up per card, refund window duration, velocity alert thresholds, terminal assignments
  • Velocity Anomaly Alerts — when a vendor's offline batch sync shows average transaction value more than 3× their online baseline, the system auto-flags the stall, freezes their payout, and surfaces an alert for manual review
  • Card Management — view all issued cards, current server balance, status (active / blocked / expired). Block or freeze any card UID instantly. This is the admin-side of the lost card freeze flow — when an attendee reports a lost card through the Companion App, the freeze request surfaces here for the admin to confirm and action.
  • End-of-Day Settlement — per-vendor payout calculation, refund deduction, payout trigger, PDF and Excel report generation

Supervisor

  • Refund Approval Queue — review each refund request raised by attendees or booth staff, approve or reject. On approval, a terminal writes the refunded balance back to the physical card.
  • Lost Card Replacement — scan a new blank card, enter the lost card UID (from the attendee's account), transfer server balance, block the old UID. Done in under 2 minutes.
  • Stall Audit — real-time full transaction list for any stall. Used during manual audits of flagged vendors.
  • Dispute Log — full history of every flagged transaction with resolution status.

Accountant

  • Read-only access to settlement reports and transaction exports.
  • Download per-event reports in PDF or Excel. Filter by stall, category, time window, or transaction type.
  • No write access — cannot approve refunds, block cards, or modify event configuration.

5. Backend Architecture

One FastAPI backend serves all three apps. It is the only system with write access to the PostgreSQL ledger. Security is enforced entirely at the API level through JWT role claims.

5.1 Tech Stack

Layer Tool Reason
Framework FastAPI (Python) Async by default. Handles hundreds of concurrent sync requests from stalls reconnecting after network outages.
Database PostgreSQL ACID-compliant. Financial data requires proper transactions, rollbacks, and complete audit logs. No NoSQL.
Cache Redis Caches card balances (TTL 30s). Acts as pub/sub broker for Socket.IO live dashboard events.
Background Jobs Celery (Redis broker) Handles offline sync batches, reconciliation, report generation, settlement payouts — all async.
Scheduled Jobs Celery Beat Auto-settle after event end, auto-archive old event data, periodic reconciliation health checks.
Auth JWT + Refresh Tokens Role encoded in token claims: admin, supervisor, booth_staff, accountant, attendee. Validated on every request.
Realtime Socket.IO Pushes live transaction events to the Admin Panel dashboard. Redis pub/sub allows horizontal scaling.
ORM SQLAlchemy + Alembic Clean async ORM models. Alembic handles version-controlled database migrations.
File Storage AWS S3 Settlement report PDFs, Excel exports. Pre-signed URLs for secure time-limited downloads.
AI Proxy Anthropic Claude API Backend proxy for attendee chatbot. Adds event context to system prompt. Attendee financial data never sent raw to Claude.

5.2 Core Endpoints — Domain Summary

Authentication /auth/*

  • POST /auth/register — attendee self-registration
  • POST /auth/login — any role. Returns JWT + refresh token. Role claim set from user record in DB.
  • POST /auth/refresh — exchange refresh token for new access token. Old token invalidated.

Cards /cards/*

  • POST /cards/link — attendee links their NFC card UID to their account
  • GET /cards/{uid}/balance — returns server balance (cached in Redis, TTL 30s)
  • GET /cards/{uid}/status — returns active / blocked / expired (used by POS before writes)
  • POST /cards/{uid}/block — admin or supervisor blocks/freezes a card. Instant propagation to online terminals.
  • POST /cards/replace — supervisor transfers balance from lost card UID to new card UID

Transactions /transactions/* and /sync/*

  • GET /transactions — paginated history, filterable by card, stall, time, type
  • POST /sync/batch — POS App posts offline transaction batches. Idempotent: duplicate txn_id is silently ignored.
  • POST /sync/force — force-flush entire SQLite queue at event close
  • Celery task on batch receipt: reconcile transaction order using on-card txn_counter, update PostgreSQL ledger, publish Socket.IO event

Top-Ups /topups/*

  • POST /topups/initiate — creates Razorpay order, returns order_id
  • POST /topups/webhook — Razorpay calls this with HMAC-signed payload. Validates signature, queues Celery task to write balance to card at next Top-Up terminal.

Refunds /refunds/*

  • POST /refunds/request — attendee raises a dispute on a specific transaction
  • POST /refunds/flag — booth staff flags a transaction for refund
  • POST /refunds/{id}/approve — supervisor approves. Queues Celery task to write refunded balance back to card.
  • POST /refunds/{id}/reject — supervisor rejects with reason.

AI Chatbot /ai/*

  • POST /ai/chat — receives attendee message, constructs a system prompt with event context (current balance, recent 10 transactions, event name, stall map summary), forwards to Claude API, returns response. Conversation history maintained per session in Redis with TTL.

Settlement /settlement/*

  • POST /settlement/generate — admin triggers. Celery calculates per-stall net (gross minus approved refunds), generates PDF and Excel, uploads to S3, emails download link.
  • POST /settlement/stalls/{id}/payout — admin approves or rejects individual stall payout. Frozen stalls (anomaly flag) require explicit approval.

Realtime /ws/*

  • Socket.IO endpoint for Admin Panel. Receives events: txn_created, alert_raised, card_blocked, sync_received

5.3 Idempotency and Reconciliation

Every transaction the POS App generates is assigned a UUID (txn_id) at the moment of writing to SQLite on the phone. When the batch is synced to the server, the endpoint checks: "Does this txn_id already exist in the ledger?" If yes, it returns 200 OK and discards the duplicate silently. This means POS Apps can retry failed sync requests indefinitely without creating phantom charges.

Transaction order reconciliation uses the on-card txn_counter. Even when three stalls sync simultaneously with interleaved timestamps, the server reconstructs the correct sequence of debits and credits for any given card by sorting by txn_counter.

6. Security Architecture

6.1 Firewall — Is it Needed?

Yes. A firewall is required, but it operates at the infrastructure level (cloud VPC security group rules), not as a separate physical appliance. The firewall's job is to ensure the database and cache are never reachable from the public internet.

What to Open and What to Close

Resource Port Public? Who Can Access Why
FastAPI (HTTPS) 443 Yes All apps, Razorpay webhook All app communication
FastAPI (HTTP) 80 Yes (redirect only) Redirect to 443 HTTPS enforcement
PostgreSQL 5432 NO FastAPI container only (VPC internal) Prevents direct DB brute-force
Redis 6379 NO FastAPI + Celery workers (VPC internal) Cache and broker not publicly accessible
Celery workers Internal NO Celery Beat + FastAPI (VPC internal) Job processing only
S3 / File storage 443 (HTTPS) Via pre-signed URLs only Authenticated users with time-limited tokens Reports download
Core firewall rule: PostgreSQL (port 5432) and Redis (port 6379) must be completely closed to the public internet. Only the FastAPI container, whitelisted by VPC security group rules, may communicate with them.

6.2 Mobile App Security

Attendee App (FizMaa Companion)

  • All communication over HTTPS (TLS 1.3 minimum)
  • JWT stored in Expo Secure Store (Keychain on iOS, Keystore on Android) — never in AsyncStorage
  • NFC only reads card UID — no write access, no balance data exposed on the attendee's phone
  • AI chat messages proxied through backend — Claude API key never in the mobile bundle

POS App (FizMaa POS) — Higher Security Requirements

  • All communication over HTTPS (TLS 1.3 minimum)
  • SSL Pinning implemented via a custom OkHttp interceptor: the app's expected server certificate hash is bundled at build time. If a man-in-the-middle proxy (Wireshark, Charles Proxy, Burp Suite) intercepts the connection, the certificate hash won't match and the connection is killed before any data is transmitted.
  • HMAC signing secret key (for computing signature over card balance sector) stored in Expo Secure Store — never in JS bundle, never in source code, injected at EAS Build time via EAS Secrets.
  • JWT scoped to booth_staff role — cannot call admin, supervisor, or attendee-exclusive endpoints.
  • Device binding (Phase 2): each device registered by IMEI/Android ID. A POS JWT will only work from the registered device.

6.3 The Four Security Problems — Solved

Problem Attack Scenario Solution
Lost card Finder taps lost card at a stall and spends the balance Attendee reports via Companion App → admin/supervisor freezes card UID in Admin Panel → online terminals block within seconds → Supervisor Booth replaces card with balance transferred
Torn write Phone battery dies mid-NFC-write, card data corrupted Two-field write-verify protocol. balance_shadow recovers the last valid state if balance_primary is corrupted mid-write.
Out-of-order sync Top-up not yet on server when payment arrives; server balance looks negative On-card txn_counter. Server holds reconciliation until all transactions for a card are received in order via Celery.
Vendor overcharges offline Stall goes to airplane mode, charges ₹1,500 instead of ₹150 Visual Handshake Screen (attendee sees amount before tapping) + velocity anomaly detection freezes payout on sync.
Double-tap charge Card read twice within 1 second, balance deducted twice 3-layer defence: app NFC lock (3s) + on-card nonce check (5s) + server idempotency key (permanent).

8. CI/CD Pipeline

The pipeline runs on GitHub Actions. Every push to any branch triggers CI. Deployment (CD) only triggers on merges to develop (staging) or main (production).

8.1 Backend CI/CD

CI Phase — triggered on every PR to develop or main

  1. Checkout code and set up Python 3.12 with pip cache
  2. Install dependencies from requirements.txt
  3. Run ruff linter — enforces code style and import ordering
  4. Run mypy type checker — ensures Pydantic model type safety
  5. Spin up temporary PostgreSQL and Redis instances via GitHub Actions service containers
  6. Run Alembic migrations against the test database
  7. Execute pytest suite — covers: idempotency tests, velocity anomaly detection, refund approval flow, JWT role enforcement, AI proxy prompt construction
  8. Check test coverage — must be above 80% or CI fails

CD Phase — triggered on merge to main

  1. Build Docker image from backend/Dockerfile
  2. Tag image with git SHA: fizmaa-backend:abc1234
  3. Push image to AWS ECR (Elastic Container Registry)
  4. Run Alembic database migrations against production PostgreSQL (inside the pipeline, before traffic switches)
  5. Deploy new task definition to AWS ECS. ECS performs a blue-green swap: new container health-checked, traffic shifted, old container terminated.
  6. Post-deploy smoke test: call GET /health on the new deployment. If it fails, ECS automatically rolls back to the previous task definition.
  7. Notify Slack #deployments with deployment summary
Zero-downtime guarantee: The ECS blue-green deployment means no in-flight sync requests from field POS devices are dropped during a deployment. The old container stays alive until all its connections drain naturally.

12. Database Documentation

PostgreSQL schema reference for the FizMaa Cashless NFC payment ecosystem · All tables, columns, relationships, and design decisions explained.

12.1 Overview

FizMaa uses a single PostgreSQL database shared across all three apps (Companion, POS, Admin Panel). The database is the only authoritative source of truth for server-side balances. The physical NFC card is the source of truth for balance in offline mode — the two stay in sync through the transaction counter reconciliation system.

The schema is managed with SQLAlchemy (async ORM) and Alembic (migration scripts). Every structural change to the database is version-controlled as an Alembic migration file.

Database Design Principles

  • ACID compliance is non-negotiable. Every rupee movement is wrapped in a database transaction with proper rollback on failure. No money can be created or destroyed by a partial failure.
  • Soft deletes everywhere. No row is ever DELETEd. Cards, stalls, users, and events are marked is_active = false or given a status field. This preserves the audit trail.
  • Immutable transaction ledger. Rows in the transactions table are never updated or deleted after they are written. Refunds and corrections are new rows, not edits.
  • All monetary amounts stored as integers (paise). ₹150.00 is stored as 15000. This eliminates floating-point rounding errors entirely.
  • All timestamps in UTC. Stored as TIMESTAMPTZ (timestamp with time zone). The application layer handles display conversion.

Entity Relationship Summary

events
  └── stalls (many per event)
        └── stall_staff (many staff per stall)

users
  └── cards (one active card per user, historically many)
        └── transactions (many per card)
              └── refunds (one refund request per transaction)

users
  └── topups (many per user)
        └── (triggers card balance write via Celery)

users
  └── ai_chat_sessions (many per user)
        └── ai_chat_messages (many per session)

events
  └── settlement_reports (one per event)
        └── stall_payouts (one per stall per report)

12.2 Table Reference

users

Stores every person who interacts with the system — attendees, booth staff, supervisors, admins, and accountants. A single role column determines what they can do. Role enforcement is applied at the API level using JWT claims derived from this field.

Column Type Constraints Description
id UUID PK, default gen_random_uuid() Primary key. UUID used instead of serial integer to prevent enumeration attacks.
email VARCHAR(255) UNIQUE, NOT NULL Login identifier. Unique across all roles.
phone VARCHAR(20) UNIQUE, NULLABLE Optional. Used for SMS notifications and walk-in attendee identification.
name VARCHAR(255) NOT NULL Display name.
password_hash TEXT NOT NULL Bcrypt hash. Raw password never stored.
role VARCHAR(20) NOT NULL, CHECK One of: attendee, booth_staff, supervisor, admin, accountant. Encoded into JWT claims on login.
is_active BOOLEAN NOT NULL, default true Soft delete flag. Deactivated users cannot log in.
is_anonymous BOOLEAN NOT NULL, default false true for walk-in attendees registered without an email. Can be upgraded to a full account via the Companion App.
created_at TIMESTAMPTZ NOT NULL, default now() Account creation time.
last_login_at TIMESTAMPTZ NULLABLE Updated on every successful login. Used to detect dormant accounts.

Indexes:

  • idx_users_email on (email) — fast login lookup
  • idx_users_role on (role) — used to list all staff for an event

Notes: Walk-in attendees who pay cash at the gate get an is_anonymous = true account created automatically by the POS App. If they later download the Companion App and tap their card, the system links their anonymous profile to their real email. Booth staff accounts are created by the admin before the event and assigned to stalls via the stall_staff table.

refresh_tokens

Stores active refresh tokens for JWT authentication. When an access token expires, the client sends its refresh token to get a new pair. The old refresh token is invalidated immediately (rotation pattern), preventing token reuse attacks.

Column Type Constraints Description
id UUID PK, default gen_random_uuid() Primary key.
user_id UUID FK → users.id, NOT NULL The user this token belongs to.
token_hash TEXT UNIQUE, NOT NULL SHA-256 hash of the raw token. Raw token only exists in the HTTP response and the client's secure store.
expires_at TIMESTAMPTZ NOT NULL Token expiry. Celery Beat purges expired rows nightly.
is_revoked BOOLEAN NOT NULL, default false Set to true on use (rotation) or on explicit logout.
created_at TIMESTAMPTZ NOT NULL, default now() Issuance time.
device_info TEXT NULLABLE Optional. Stores device identifier for the token (e.g. Android 14, Pixel 7). Useful for the device binding feature in Phase 2.

events

Represents a single festival or event. All stalls, cards, transactions, and settlement reports are scoped to an event. This is the top-level entity in the schema.

Column Type Constraints Description
id UUID PK, default gen_random_uuid() Primary key.
name VARCHAR(255) NOT NULL Human-readable name, e.g. Sunburn Festival 2026.
slug VARCHAR(100) UNIQUE, NOT NULL URL-safe identifier, e.g. sunburn-2026. Used in API paths.
venue VARCHAR(255) NOT NULL Venue name and city, e.g. Vagator Beach, Goa.
starts_at TIMESTAMPTZ NOT NULL Event start time.
ends_at TIMESTAMPTZ NOT NULL Event end time. Celery Beat uses this to trigger auto-settlement.
status VARCHAR(20) NOT NULL, default draft One of: draft, active, ended, settled. Controls what operations are permitted.
max_topup_per_card INTEGER NOT NULL, default 500000 Maximum lifetime top-up allowed per card for this event, in paise. Default ₹5,000.
refund_window_minutes INTEGER NOT NULL, default 30 Refund requests older than this many minutes are automatically rejected.
velocity_alert_multiplier NUMERIC(4,2) NOT NULL, default 3.0 Offline batch average transaction value above this multiple of online baseline triggers a velocity alert.
organizer_id UUID FK → users.id, NOT NULL The admin user who created and owns this event.
created_at TIMESTAMPTZ NOT NULL, default now() Creation time.

stalls

Represents a vendor stall operating at an event. Each stall has at least one staff member assigned to it. Revenue, refunds, and payouts are calculated per stall at settlement.

Column Type Constraints Description
id UUID PK, default gen_random_uuid() Primary key.
event_id UUID FK → events.id, NOT NULL The event this stall operates at.
name VARCHAR(255) NOT NULL Stall name, e.g. Biryani Bros.
code VARCHAR(20) NOT NULL Short location code, e.g. B04. Printed on the stall's physical sign.
category VARCHAR(50) NOT NULL One of: food, beverages, merch, other. Used for spending insights and settlement grouping.
is_active BOOLEAN NOT NULL, default true Soft delete. Deactivated stalls stop appearing in the admin dashboard.
gross_revenue_paise INTEGER NOT NULL, default 0 Running total of all completed transactions for this stall, in paise. Updated by Celery on each sync batch. Denormalized for fast dashboard queries.
created_at TIMESTAMPTZ NOT NULL, default now() Creation time.

Indexes:

  • idx_stalls_event_id on (event_id) — fast stall listing per event
  • idx_stalls_event_category on (event_id, category) — category breakdown queries

Notes: gross_revenue_paise is a denormalized cache column. It is always re-computable from SUM(transactions.amount_paise) WHERE stall_id = X AND status = 'completed'. It exists to avoid expensive aggregation queries on the hot dashboard path.

stall_staff

Join table assigning users with the booth_staff role to specific stalls. A staff member can work at multiple stalls (though unusual in practice), and a stall can have multiple staff members for shift coverage.

Column Type Constraints Description
id UUID PK, default gen_random_uuid() Primary key.
stall_id UUID FK → stalls.id, NOT NULL The stall this assignment is for.
user_id UUID FK → users.id, NOT NULL The staff member being assigned. Must have role = 'booth_staff'.
assigned_at TIMESTAMPTZ NOT NULL, default now() When the admin created this assignment.
assigned_by UUID FK → users.id, NOT NULL The admin user who made the assignment.

Constraints:

  • UNIQUE (stall_id, user_id) — prevents duplicate assignments

Notes: The stall_id assigned to a staff member is encoded in their JWT at login time. The POS App reads this from the token to know which stall to attribute transactions to. This means if a staff member moves stalls mid-event, they must log out and log back in to get a new token.

cards

Represents a physical MIFARE NFC card issued to an attendee. The card_uid is the permanent, unalterable hardware identifier read from the chip. The server_balance_paise is the backend's record of what should be on the card — it is updated on every confirmed transaction and top-up.

Column Type Constraints Description
id UUID PK, default gen_random_uuid() Primary key.
card_uid VARCHAR(50) UNIQUE, NOT NULL The hardware UID of the NFC card, e.g. A8:F2:3D:91:C6:B4. Set once, never changed.
event_id UUID FK → events.id, NOT NULL The event this card was issued for. Cards are event-scoped.
user_id UUID FK → users.id, NULLABLE The attendee this card is linked to. NULL for unprovisioned cards that have been written but not yet linked to a Companion App account.
status VARCHAR(20) NOT NULL, default active One of: active, blocked, replaced, expired.
server_balance_paise INTEGER NOT NULL, default 0 Backend's record of the card balance, in paise. Must stay in sync with the on-card balance through the reconciliation system.
txn_counter INTEGER NOT NULL, default 0 The last known on-card transaction counter value. Updated every time a transaction for this card is reconciled. Used to detect missed syncs.
card_type VARCHAR(30) NOT NULL, default mifare_ultralight One of: mifare_ultralight, mifare_classic_1k. Determines which read/write protocol the POS App uses.
tier VARCHAR(20) NOT NULL, default general One of: general, gold, platinum, vip. Stored server-side since it cannot fit on Ultralight cards. Used by online POS terminals to apply tier-based discounts.
issued_at TIMESTAMPTZ NOT NULL, default now() When the card was provisioned at the gate.
linked_at TIMESTAMPTZ NULLABLE When the attendee linked the card to their Companion App account. NULL for walk-in attendees who never downloaded the app.
blocked_at TIMESTAMPTZ NULLABLE Timestamp of the block action. Populated when status becomes blocked.
blocked_by UUID FK → users.id, NULLABLE The admin or supervisor who blocked the card.
replaced_by_card_id UUID FK → cards.id, NULLABLE If this card was replaced (lost card flow), this points to the new card. Preserves the audit chain.

Indexes:

  • idx_cards_card_uid on (card_uid) — the most common lookup (every POS read and attendee balance check)
  • idx_cards_user_id on (user_id) — attendee's card history
  • idx_cards_event_id_status on (event_id, status) — admin card management view

Notes: A user can have multiple cards in history (e.g. after a lost card replacement), but only one should be status = 'active' per event at any time. This is enforced at the application layer, not with a unique constraint, to allow for the brief window during a card replacement where both old and new cards may be technically active. server_balance_paise is the reconciled balance after all synced transactions are applied. During an offline period, the on-card balance and the server balance will diverge. The reconciliation Celery task brings them back in sync after the POS App reconnects.

transactions

The core financial ledger. Every debit (payment at a stall), credit (top-up write-back), and refund credit is recorded here as an immutable row. Rows in this table are never updated or deleted.

Column Type Constraints Description
id UUID PK, default gen_random_uuid() Primary key (server-generated).
txn_id UUID UNIQUE, NOT NULL The idempotency key. Generated by the POS App at the moment of writing to SQLite on the device, before any network communication. Used to deduplicate sync batches.
card_id UUID FK → cards.id, NOT NULL The card this transaction is against.
stall_id UUID FK → stalls.id, NULLABLE The stall that charged the card. NULL for top-up credits and system adjustments.
event_id UUID FK → events.id, NOT NULL Denormalized for fast event-scoped queries without joins.
type VARCHAR(20) NOT NULL One of: payment, topup_credit, refund_credit, adjustment.
amount_paise INTEGER NOT NULL Transaction amount in paise. Always positive. The type field determines whether this is a debit or credit.
balance_before_paise INTEGER NOT NULL Server balance on the card immediately before this transaction. Used to reconstruct balance history and for dispute resolution.
balance_after_paise INTEGER NOT NULL Server balance on the card immediately after this transaction.
status VARCHAR(20) NOT NULL, default pending One of: pending (in SQLite queue, not yet synced), synced (received by server, awaiting reconciliation), completed (reconciled and applied to the ledger), failed (reconciliation error).
txn_counter_on_card INTEGER NOT NULL The value of the card's txn_counter at the time of this transaction. Used by the reconciliation engine to order transactions correctly.
nonce VARCHAR(50) NOT NULL The last_txn_nonce value written to the card for this transaction. Stored for audit purposes.
device_id VARCHAR(100) NULLABLE The Android or iOS device ID of the POS terminal that processed this transaction. Useful for fraud investigation.
synced_at TIMESTAMPTZ NULLABLE When the POS App sent this transaction to the server. NULL while still in the device's SQLite queue.
reconciled_at TIMESTAMPTZ NULLABLE When the Celery reconciliation task applied this transaction to the ledger.
created_at TIMESTAMPTZ NOT NULL, default now() When the transaction was created on the device (moment of NFC write).

Indexes:

  • idx_txn_txn_id on (txn_id) — idempotency check on every sync batch (most critical index)
  • idx_txn_card_id on (card_id) — attendee transaction history
  • idx_txn_stall_id_event_id on (stall_id, event_id) — stall audit view
  • idx_txn_event_id_status on (event_id, status) — dashboard aggregations
  • idx_txn_created_at on (created_at DESC) — chronological listing

Notes: The combination of txn_counter_on_card and card_id is what the reconciliation engine uses to order payments. If three transactions arrive with counters 5, 7, 6, the engine re-sorts them to 5, 6, 7 and applies them in that order. If counter 6 never arrives (a missing sync), the engine puts transactions 7+ in a pending_reconciliation state until 6 arrives or an admin manually resolves the gap. balance_before_paise and balance_after_paise are denormalized at write time and never recomputed. They create a verifiable chain: balance_after of transaction N should equal balance_before of transaction N+1 for the same card.

topups

12.3 Key Relationships Explained

Card ↔ User (Many-to-One, with history)

A user can have many cards over time (one card per event, or a replacement card after loss). But at any given point, only one card should be status = 'active' per event per user. The user_id on cards may be NULL for unprovisioned walk-in cards that have been written but not yet claimed through the Companion App.

Transaction ↔ Card (Many-to-One, immutable)

Every transaction is permanently associated with a card at creation time. Transactions cannot be reassigned to a different card. The balance_before_paise and balance_after_paise columns form a verifiable chain of custody for every rupee on the card.

Transaction idempotency (txn_id)

The txn_id column on transactions is generated by the POS App at the exact moment the NFC write succeeds — before any network call. It is the idempotency key used by POST /sync/batch. The database enforces uniqueness on this column, so even if the same transaction is submitted in 10 different sync batches, only one row is ever written to the ledger.

Topup ↔ Transaction (One-to-One, deferred)

A topup row is created when the Razorpay payment is initiated. The linked transaction row (write_transaction_id) is only created later, when the attendee physically taps their card at a Top-Up terminal and the Celery task writes the balance. This two-step structure means a paid-but-not-yet-written top-up has a topup row with status = 'write_pending' but no corresponding transaction row yet.

Refund ↔ Transaction (One-to-One request, One-to-One credit)

A refund request (refunds table) references the original transaction being disputed. On approval, a second transactions row of type = 'refund_credit' is created and linked back via refunds.credit_transaction_id. The original transaction is never modified.

12.4 Alembic Migration Notes

Every schema change ships as an Alembic migration script in backend/alembic/versions/. Migrations run automatically as part of the CI/CD pipeline before the new Docker container receives any traffic.

Migration naming convention:

{timestamp}_{short_description}.py
# e.g.
20260615_001_create_users_and_auth_tables.py
20260620_002_add_cards_and_transactions.py
20260625_003_add_velocity_alerts.py
20260628_004_add_ai_chat_tables.py

Key rules:

  • Every migration must have both an upgrade() and a downgrade() function.
  • Never drop a column directly — add a new column, migrate data, deprecate old column in a subsequent migration.
  • Index creation uses CREATE INDEX CONCURRENTLY in production migrations to avoid table locks.
  • All migrations are tested against the test database in the CI pipeline before any production deploy is permitted.
