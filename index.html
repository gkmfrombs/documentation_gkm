System Overview

FizMaa Cashless is a three-app NFC payment ecosystem designed for high-density live events (music festivals, college fests, stadium food courts). To eliminate physical card corruption ("torn writes") and ensure universal iOS compatibility, Phase 1 shifts away from legacy smart-wallet memory writing. Instead, physical NTAG216 and MIFARE Ultralight EV2 wristbands act strictly as read-only Cryptographic Identity Tokens (~10ms scan speed).

All financial balances and ledger transactions are processed atomically in PostgreSQL. To guarantee sub-10ms checkout speeds and 100% uptime during cellular network blackouts inside crowded venues, POS terminals process payments over local food court Wi-Fi against an on-site Local Event Edge Server, which asynchronously batches ledger records to the AWS cloud.

1. System Architecture

All three applications and venue edge servers share a unified data contract. Role-based access control (RBAC) is enforced strictly server-side via JWT claims. By separating the hardware identity from the financial ledger, the system eliminates chip memory corruption while enabling real-time financial auditing.

1.1 Data Flow & Local Edge Resilience

  • Gate Onboarding: Volunteer scans attendee's digital QR ticket using Android POS → Taps blank NTAG216 wristband → Reads 7-byte hardware UID in 10ms → Backend binds UID to ticket profile.
  • Online Top-Up: Attendee initiates Razorpay UPI payment in Companion App → Razorpay webhook fires to cloud backend → Backend verifies HMAC signature and updates integer database balance (`server_balance_paise`). No physical card write required.
  • Stall Checkout (Edge Mode): Vendor enters amount on POS phone → Attendee taps wristband → POS reads UID + Token (~10ms) → Sends HTTP POST over local Wi-Fi router to venue Local Edge Server → Edge server verifies balance, atomically deducts funds in local Postgres, and returns 200 OK → POS flashes green.
  • Cloud Synchronization: Local Edge Server buffers completed transactions and streams them asynchronously to the AWS Neon cloud master database in optimized batch uploads every 30 seconds.

1.2 Physical Card Specs — NTAG216 / MIFARE Ultralight EV2

To support native iOS reading on customer iPhones (CoreNFC without Apple entitlements) and prevent write-tearing, we explicitly avoid MIFARE Classic 1K. Cards are pre-encoded at the factory or gate with read-only identity structures:

Field Purpose Storage Location
card_uid Permanent 7-byte hardware identifier read directly from chip silicon (unalterable). Chip Header (Read-Only)
card_token_hash SHA-256 cryptographic token validating that the wristband belongs to the official FizMaa event batch. NFC User Memory Sector
card_qr_code Printed QR code on the wristband surface matching the UID. Serves as an instant camera scanner fallback if NFC chip is stepped on or damaged. Physical Print on Card Surface
card_public_id Human-readable serial number (e.g., FZ-2026-8891) printed on the band for visual kiosk checks and customer support queries. Physical Print on Card Surface
Why Read-Only Identity Tokens?
Writing data sectors to an NFC chip takes 100–300 milliseconds. In crowded festival lines, attendees frequently pull their hands away too quickly, causing a "torn write" that corrupts on-card balance memory and freezes the wristband. Reading a UID and token string takes only 10 milliseconds. By shifting balance calculations to high-speed server databases, we achieve 100% transaction reliability and zero card corruption.

1.3 Card Provisioning & Top-Up Process

Scenario A: The Pre-Bought Ticket (Gold, Platinum, VIP)

These attendees purchased tickets online in advance, including pre-loaded balances (e.g., ₹2,000 bar credit).

  1. Verify Ticket: Attendee presents digital ticket QR code at the entrance gate. Volunteer scans QR using FizMaa POS (Registration Mode). Backend validates ticket authenticity.
  2. Physical Handover: Volunteer picks up a blank NTAG216 wristband and taps it against the back of their Android POS phone.
  3. The 10ms Link: POS reads the chip's `card_uid` in 10ms and transmits the link request to the server: "Bind UID 04:A2:B3:C4 to Ticket #9876 with starting balance ₹2,000."
  4. Instant Entry: Server confirms activation. Attendee puts on the wristband and walks directly into the venue to spend immediately.

Scenario B: The On-the-Spot Gate Walk-In

Attendees arriving without tickets who need admission and digital wallet loading simultaneously.

  1. Take Payment: Attendee pays cash or scans a static UPI QR at the registration desk for entry + ₹1,000 starting balance.
  2. Configure Top-Up: Staff enters "₹1,000" into FizMaa POS (Top-Up Mode) and taps a blank wristband.
  3. Anonymous Profile Provisioning: POS pings the Local Edge Server, instantly creating an anonymous user record (`is_anonymous = true`) bound to `card_uid` with `server_balance_paise = 100000`.
  4. Optional Companion Link: Once inside, the attendee can download the Companion App, tap their wristband to their own smartphone, and verify their mobile number via OTP to upgrade their anonymous wallet to a personal profile.

⚠️ VIP Tier Handling in v4.0

Because NTAG216 chips act as lightweight read-only tokens, tier privileges (Gold, VIP, Backstage) are not stored on silicon memory. They are managed through two operational layers:

  • Physical Visuals: Different colored wristbands (e.g., Black bands for VIP, White bands for General Admission) allow security guards to manage physical zone access instantly without scanning.
  • Online Terminal Lookup: When tapping at a VIP bar for discounts, the vendor's POS terminal queries the Edge Server during the 10ms read, identifying the user's tier from the database record and applying discounts automatically before deducting funds.

3. App 2 — FizMaa POS (Booth Staff Terminal — Android Only)

One-line purpose: Transform standard Android smartphones into high-speed, offline-resilient checkout terminals that process wristband checkouts in under 15 milliseconds over local venue Wi-Fi.

3.1 Frontend Architecture & Why Android Only?

In Phase 1, the POS application is deployed strictly on **Android smartphones**. iOS is intentionally excluded for vendor checkout terminals for three strategic reasons:

  1. Low-Level Hardware Reliability: Android provides direct, uninterrupted access to the NFC radio transmitter, allowing sub-15ms background polling without triggering system system UI popups on every tap.
  2. Bypassing App Store Delays: Vendor POS apps require rapid, same-day distribution to hundreds of volunteer devices at the venue gate. By generating standalone Android APKs via EAS Build, we distribute the app via MDM or QR download portals, bypassing Apple's lengthy enterprise review bottlenecks.
  3. Cost-Effective Hardware Scaling: Organizers can deploy fleets of standardized, ruggedized ₹10,000 Android devices across all food courts without managing expensive iOS hardware inventory.

3.2 The Checkout Handshake & Visual Anti-Fraud UI

To prevent silent overcharging by rogue booth staff—the most common fraud vector in festival cashless systems—the POS app enforces a strict visual workflow before NFC scanning is enabled:

  1. Keypad Entry: Staff types the order amount (e.g., ₹250) on a high-contrast numeric keypad and taps "Charge".
  2. The Visual Handshake Screen: The app screen transitions to a bold, full-screen confirmation displaying: **Stall Name**, **Deduction Amount (₹250)**, and an animated prompt reading *"Tap Customer Wristband Now"*. The vendor **cannot** skip or dismiss this screen. The attendee physically sees the exact charge on the phone screen before presenting their hand.
  3. The 10ms Scan: Attendee taps their wristband against the back of the phone. The app reads `card_uid` and `card_token_hash` in 10 milliseconds.
  4. Local Wi-Fi Execution: The POS app fires an HTTP POST request over the venue's local Wi-Fi router directly to the **Local Event Edge Server** running in the food court.
  5. Atomic Deduction: The Edge Server verifies the token, checks `server_balance_paise >= 25000`, deducts the amount inside an isolated PostgreSQL transaction, and returns a 200 OK confirmation.
  6. Green Success UI: The POS screen flashes vibrant green, emits a distinct success beep, and displays the customer's remaining balance.

3.3 Edge Execution & Offline Fallback

During normal festival operations, all checkouts route over local Wi-Fi to the venue's Local Edge Server, bypassing public cellular networks entirely. However, if a Wi-Fi router loses power or disconnected cables isolate a food stall, the POS terminal automatically shifts to **Emergency Offline Mode (`source = offline_emergency`)**:

  • When Wi-Fi pings fail, the app reads the wristband UID and verifies the cryptographic `card_token_hash` locally against its cached event batch secret.
  • The transaction is written immediately to an encrypted local `expo-sqlite` queue on the Android device.
  • To limit organizer financial liability during total network blackouts, offline mode enforces a strict spending cap (e.g., maximum ₹500 per tap, maximum 5 offline taps per wristband).
  • The moment Wi-Fi connectivity is restored, background sync workers flush the SQLite queue to the Edge Server, which reconciles the ledger and streams the records to the AWS cloud.

5.2 Core Endpoints & Distributed Edge Network

During a live festival, cellular internet connections inside stadiums collapse due to network congestion from thousands of attendee smartphones. If POS terminals relied on direct cloud APIs, checkouts would timeout, causing massive food stall queues.

To solve this, FizMaa deploys **Local Event Edge Servers** (compact laptops or mini-PCs running dockerized FastAPI + local Postgres) connected directly to the food court's local Wi-Fi router:

  • Local Checkout Loop: POS terminals connect to the food court Wi-Fi and send payment requests (`POST /api/v1/edge/transactions`) to the Local Edge Server. Latency is sub-10ms because packets never leave the local venue network.
  • Atomic Database Isolation: The Edge Server executes an isolated PostgreSQL transaction using row-level locking (`SELECT ... FOR UPDATE`). It checks the card's local balance, subtracts the funds atomically, inserts the ledger record, and returns success.
  • Asynchronous Upstream Sync: Every 30 seconds, a background worker on the Edge Server bundles newly completed transactions into a `SyncBatch` payload and securely streams them up to the AWS Neon cloud database to keep attendee mobile apps synchronized.

5.3 Idempotency & Ledger Reconciliation

In distributed networks, Wi-Fi packet drops can cause a POS terminal to retry sending a checkout request that the server already processed. Without strict safeguards, this would double-bill the attendee.

FizMaa eliminates duplicate billing through strict **Idempotency Key Architecture**:

  1. When a vendor taps "Charge" on the POS app, the app generates a unique UUID string (`idempotency_key`, e.g., `chk_stall4_1142am_88a1`) stored in memory before initiating NFC scanning.
  2. Every network request dispatched to the Edge Server or Cloud carries this exact idempotency key.
  3. The `transactions` table in PostgreSQL enforces a strict database-level `UNIQUE` index on `idempotency_key`.
  4. If a network flicker forces the POS terminal to retry the upload, the database encounters a unique constraint violation. The backend service layer catches this exception, ignores the duplicate deduction, and safely returns the existing `200 OK` receipt from the original transaction. Money is only deducted once.

To protect financial ledgers from external network attacks, database and cache ports are strictly isolated inside an AWS Virtual Private Cloud (VPC). Direct public internet access to storage engines is permanently blocked.

6.2 Mobile App Security Hardening

Companion App (Consumer Security):

  • All network communication enforced over HTTPS (TLS 1.3 minimum). Unencrypted HTTP connections are rejected by OS transport security policies.
  • User authentication JWTs and Refresh Tokens are stored natively inside **Expo Secure Store** (iOS Keychain / Android Keystore encrypted hardware enclaves). Standard `AsyncStorage` is strictly forbidden for token storage.
  • NFC hardware interactions are restricted programmatically to read-only scanning (`NFCMiFareTag`). The mobile binary contains zero code capable of modifying chip memory sectors.

POS App (Vendor Terminal Hardening):

  • SSL Certificate Pinning: Implemented via custom OkHttp network interceptors. The cryptographic hash of our backend TLS certificate is bundled directly into the Android binary. If a malicious user attempts man-in-the-middle packet sniffing using Wireshark or Charles Proxy, the certificate hash mismatch kills the connection immediately before authentication headers are transmitted.
  • EAS Secrets Injection: Cryptographic batch verification hashes and API routing secrets are injected at compile time via Expo EAS Secrets. Sensitive keys never exist in plaintext within visual source code repositories.
  • Device Binding (Phase 2 Roadmap): Each POS terminal IMEI or Android ID is bound to a specific vendor user account upon initial login. If stolen credentials are used on an unauthorized device, the backend rejects the JWT signature.

12. Database Documentation

12.1 Overview & Architectural Design Principles

FizMaa uses a unified PostgreSQL relational database schema deployed across the AWS Cloud master instance and localized venue Edge Servers. The cloud database is the **authoritative source of truth for all financial ledgers and user balances**. Physical NFC wristbands act purely as cryptographic identity pointers.

Core Database Design Principles

  • 100% Integer Paise Accounting: To permanently eliminate floating-point rounding errors (`₹149.999999`), all monetary balances, transaction values, top-ups, and refunds are stored strictly as integers in **paise** (`₹1.00` = `100 paise`; `₹500.00` = `50000`). Decimal and float SQL data types are strictly forbidden.
  • Universal Soft Deletes: No record is ever deleted via SQL `DELETE`. Entities use `is_active = false` or explicit lifecycle status enums (`LOST`, `REVOKED`, `DECOMMISSIONED`) to maintain complete audit traceability for tax and legal compliance.
  • Immutable Append-Only Ledger: Rows in the `transactions` table represent financial history and are **never modified or updated after insertion**. Refunds, error corrections, and balance adjustments are inserted as brand new credit/debit rows referencing the original transaction.
  • Surrogate UUID Primary Keys: All tables utilize database-generated UUIDs (`gen_random_uuid()`) as primary keys (`id`). Hardware serials (like chip strings `card_uid`) are stored as unique indexed alternate keys. This ensures that if a user loses their physical wristband, a replacement card can be linked to their account without corrupting historical foreign key transaction records.
  • Universal UTC Timestamps: All temporal data is stored as `TIMESTAMPTZ` (Timestamp with Time Zone in UTC). Application client presentation layers handle local time zone conversions (e.g., IST `+05:30`) dynamically at rendering time.

12.3 Key Relationships & System Rules Explained

Card ↔ User (One-to-Many Historical; One-to-One Active)

Over the course of a festival season, an attendee (`users` row) can be linked to multiple `cards` rows historically (e.g., if their original wristband was lost and replaced, or if they attend multiple festivals over the year). However, application business logic strictly enforces that an attendee can have **at most one active wristband (`status = ACTIVE`) per festival event** at any given moment.

Transaction ↔ Card (Many-to-One Immutable Binding)

Every checkout record in `transactions` is permanently bound to a `card_id` at insertion time. This binding cannot be modified or re-assigned. The denormalized `balance_before_paise` and `balance_after_paise` columns create a mathematically verifiable chain of custody: for any given wristband, `balance_after_paise` of Transaction $N$ must exactly equal `balance_before_paise` of Transaction $N+1$. Any mathematical discrepancy alerts accounting immediately.

Topup ↔ Transaction (Two-Step Deferred Execution)

When an attendee initiates a Razorpay payment in the Companion App, a row is created in `topups` (`status = initiated`). At this stage, zero balance is added to the wallet. Once the official Razorpay webhook verifies payment clearance (`status = paid`), a Celery background worker executes a database transaction: it increments `cards.server_balance_paise`, inserts an immutable credit row into `transactions` (`type = topup`), and binds the ledger row back to the top-up via `topups.credit_transaction_id`.

Refund ↔ Transaction (Strict 1-to-1 Claim Binding)

To prevent double-dipping disputes, the `refunds` table enforces a unique foreign key constraint on `transaction_id`. Only one dispute claim can ever be initiated against a purchase. When a supervisor approves the claim, the original purchase row remains untouched; the system inserts a fresh credit row into `transactions` (`type = refund`) and links it via `refunds.credit_transaction_id`.

12.4 Alembic Migration Protocol

All database structural changes (creating tables, altering columns, adding indexes) must ship as version-controlled Python migration scripts inside `backend/alembic/versions/`. Modifying production database schemas manually via SQL consoles is strictly prohibited.

Migration File Naming Convention:

{year}{month}{day}_{revision_id}_{short_description}.py
# Example:
20260701_a560bd3cc74e_add_v4_edge_and_identity_tables.py
20260715_b891ca4ff120_add_device_registrations_table.py

Mandatory Engineering Rules for Migrations:

  • Bidirectional Execution: Every migration script must define both an `upgrade()` function (to apply structural changes) and a corresponding `downgrade()` function (to revert cleanly if a deployment fails).
  • Order of Operations for Drops: When removing legacy columns or tables (such as stripping old smart-wallet string `uid` columns), migration scripts must explicitly drop dependent foreign key constraints (`op.drop_constraint(...)`) **before** executing column drop operations (`op.drop_column(...)`).
  • Zero-Downtime Indexing: When adding indexes to high-volume tables (`transactions`, `cards`) in production, scripts must utilize concurrent indexing (`postgresql_concurrently=True`) to prevent exclusive table locks from stalling active POS checkout traffic.
  • Automated CI Verification: GitHub Actions CI pipelines automatically execute `alembic upgrade head` followed by `alembic downgrade base` against ephemeral Docker PostgreSQL containers on every pull request. PRs with broken migration chains are blocked from merging.
