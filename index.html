FizMaa Cashless Ecosystem

A high-speed, offline-resilient NFC payment system for festivals. Built on read-only Identity Tokens, Local Edge Servers, and atomic cloud ledger synchronization.

Phase 1: Active Development POS: Android Only Companion: iOS + Android Hardware: NTAG216 / Ultralight EV2

System Overview

FizMaa Cashless replaces cash at live events (festivals, college fests, food expos) with NTAG216 and MIFARE Ultralight EV2 smart cards. To eliminate physical card corruption (write-tearing) and ensure universal iOS compatibility, Phase 1 treats the NFC chip as a read-only Cryptographic Identity Token. All financial transactions occur atomically on a Local Event Edge Server (over venue Wi-Fi) or the AWS Cloud backend.

1. System Architecture

The system decouples identity from balance. The physical NFC card holds only a permanent UID and a cryptographic token hash. The actual rupee balance resides strictly in PostgreSQL tables managed by the cloud backend or the Local Event Edge Server.

1.1 Data Flow & Local Edge Resilience

  • Online Top-Up: Attendee pays via Razorpay inside the Companion App → Razorpay Webhook fires to Cloud Backend → Backend verifies HMAC signature and updates `server_balance_paise`. (No physical card write required for top-ups).
  • Stall Checkout (Edge Mode): POS App (Android) taps attendee card → Reads UID + Token (~10ms) → Sends HTTP POST over venue Wi-Fi to Local Edge Server → Edge Server verifies balance, atomically deducts funds, and logs transaction → POS displays visual success.
  • Cloud Synchronization: Local Edge Server asynchronously batches completed transactions to the AWS Neon Postgres cloud database every 30 seconds via background workers.

1.2 Physical Card Specs — NTAG216 / MIFARE Ultralight EV2

To support native iOS reading on customer phones and prevent mid-write data corruption, we explicitly avoid MIFARE Classic 1K. Cards are pre-encoded at the factory or gate with read-only identity structures:

Field Purpose Storage Location
card_uid Permanent 7-byte hardware identifier read from the silicon chip. Chip Header (Read-Only)
card_token_hash SHA-256 cryptographic token validating that the card belongs to the official FizMaa event batch. NFC User Memory Sector
card_qr_code Printed QR code on the physical card surface matching the UID. Serves as an instant camera fallback if NFC chip fails. Physical Print on Card Surface
card_public_id Human-readable serial number (e.g., FZ-2026-8891) printed on the card for support queries. Physical Print on Card Surface
Why Read-Only Identity Tokens?
Writing data sectors to an NFC chip takes 100–300ms. In crowded festival lines, attendees frequently pull their wristbands away too quickly, causing a "torn write" that corrupts on-card balance memory. Reading a UID/Token takes only 10ms. By shifting balance math to database software, we achieve 100% transaction reliability and zero card corruption.

1.3 Provisioning & Top-Up Flows

Scenario A: Pre-Bought Ticket Linking

  1. Attendee presents digital ticket QR code at the entrance gate.
  2. Staff scans ticket QR using **FizMaa POS (Registration Mode)**.
  3. Staff taps a fresh NTAG216/Ultralight wristband to their Android phone.
  4. POS App reads `card_uid` and links it to the ticket's pre-loaded balance in the backend/edge server. The attendee enters immediately.

Scenario B: Gate Walk-In Cash/UPI Top-Up

  1. Attendee pays physical cash or scans a static UPI QR at the booth.
  2. Staff enters amount (e.g., ₹1,000) on **FizMaa POS (Top-Up Mode)** and taps a blank wristband.
  3. POS App pings the Edge Server to create an anonymous user profile linked to `card_uid` with a starting balance of ₹1,000.
  4. Optional: Attendee downloads Companion App later, taps their card, and links the anonymous balance to their phone number.

2. Companion App (iOS + Android)

Purpose: A consumer-facing mobile application allowing attendees to safely manage their funds without ever touching booth payment deduction flows.

2.1 Core Capabilities (Safe Actions Only)

  • NFC Card Linking: Uses Apple CoreNFC (iOS) and Android NFC to perform a 10ms read-only scan of the wristband, permanently linking it to the user's account.
  • Webhook-Verified Top-Ups: Attendees add funds via Razorpay (UPI/Cards). To prevent client-side spoofing, the app balance updates **only** after the backend receives and validates the Razorpay webhook.
  • Live Balance & History: Fetches running balance and itemized expenditure receipts directly from the cloud backend.
  • Instant Kill-Switch: Allows attendees to freeze a lost wristband immediately. The block flag propagates to edge servers within seconds, declining any subsequent tap attempts at stalls.

3. POS App (Android Only — Phase 1)

Purpose: Turn vendor Android smartphones into high-speed, offline-resilient checkout terminals. iOS is explicitly excluded in Phase 1 to guarantee low-level hardware reliability and eliminate App Store deployment barriers.

3.1 The Checkout Handshake & Edge Execution

  1. Staff enters rupee amount on the large keypad and hits "Charge".
  2. Visual Handshake Screen: App displays a full-screen confirmation showing Deduction Amount and Stall Name. Vendor cannot skip this screen, preventing silent overcharging.
  3. Attendee taps wristband. POS reads `card_uid` + `card_token_hash` in ~10ms.
  4. POS sends transaction payload to the **Local Event Edge Server** over venue Wi-Fi.
  5. Edge Server validates identity, checks for duplicate nonces (5-second window), deducts balance atomically in Postgres, and returns `200 OK`.
  6. POS screen flashes green with a success beep. If Wi-Fi drops completely, POS falls back to an encrypted local SQLite queue (`emergency_offline` mode) with strict liability limits.

4. Admin Panel (Web)

Purpose: Mission control for event organizers, supervisors, and accountants. Built with Next.js App Router and Tailwind CSS.

4.1 Role-Gated Operational Views

  • Live Event Dashboard: Real-time Socket.IO feed displaying gross velocity, active cards, stall revenue leaderboards, and Edge Server sync health.
  • Velocity Anomaly Detection: Automatically freezes vendor payouts if an offline/edge sync batch shows average transaction values exceeding 3× their historical baseline.
  • Supervisor Dispute Queue: Review attendee refund requests. Approved refunds instantly credit the user's cloud balance without requiring a physical card tap.
  • Lost Card Replacement: Supervisor scans a new blank card, enters the lost card's serial number, and atomically migrates the balance while permanently revoking the old UID.

5. Backend & Local Edge Server Architecture

The system utilizes a dual-tier backend model: a highly available AWS Cloud API (FastAPI + Neon Postgres) paired with deployable Local Edge Servers for venue network isolation.

5.1 Tech Stack & Infrastructure

Layer Technology Architectural Role
API Framework FastAPI (Python 3.12) Asynchronous handling of high-concurrency sync spikes and POS checkouts.
Primary Database PostgreSQL (Neon Cloud) ACID-compliant master ledger. Employs connection pooling (`pool_pre_ping=True`).
Edge Database PostgreSQL (Dockerized local) Runs on venue laptops/mini-PCs. Handles sub-10ms local checkouts during network drops.
Background Jobs Celery + Redis Broker Processes asynchronous edge sync batches, PDF settlement generation, and webhook queues.
Realtime Push Socket.IO + Redis Pub/Sub Streams live transaction events and fraud alerts to the Admin Panel.

6. Security Architecture

6.1 The Four Event Security Problems — Solved

Threat Vector Attack Scenario FizMaa Architectural Solution
Torn Write Corruption Attendee pulls card away mid-write; chip balance sectors become readable/corrupted. Eliminated by Design. Chips are read-only (~10ms scan). All balance mutations execute atomically inside PostgreSQL transactions.
Lost Card Spending Finder taps a dropped wristband at stalls to spend remaining funds. Attendee clicks "Freeze" in Companion App → Cloud pushes revocation flag to Edge Servers → Terminal rejects card on next tap.
Client-Side Top-Up Spoofing Malicious user intercepts mobile network response to fake a successful UPI payment. Top-ups are strictly gated by **Razorpay Webhook verification** (`POST /topups/webhook`) with server-side HMAC signature validation.
Vendor Offline Overcharging Stall goes disconnected and bills ₹1,500 instead of ₹150 for an item. Mandatory Visual Handshake Screen + backend **Velocity Anomaly ML algorithm** auto-freezes vendor settlement payouts upon batch sync.

8. CI/CD Pipeline

  • Backend CI/CD (GitHub Actions): Runs `ruff` linting, `mypy` strict type-checking, and executes `pytest` against ephemeral Postgres containers. Auto-deploys to AWS ECS on `main` merge using zero-downtime blue-green tasks.
  • POS Mobile Build (EAS Build): Automatically compiles Android APKs/AABs on release tags. Builds are distributed directly to vendor devices via MDM or staff download portals (Not published to public App Stores).

9. Deployment Process

Cloud infrastructure is managed via Terraform on AWS (ECS Fargate, RDS Postgres, ElastiCache Redis). For field deployment, the **Local Edge Server** is packaged as a single `docker-compose.edge.yml` file. Venue IT engineers execute one command on a local mini-PC connected to the food court Wi-Fi router, instantly establishing a resilient, low-latency POS checkout network.

12. Database Reference

The system utilizes an immutable, ACID-compliant relational schema. To support Phase 1 architectural upgrades, the database explicitly models read-only identity tokens, multi-source transaction logging, and local edge server synchronization batches.

12.1 Design Principles

  • Integer Monetary Accounting: All balances, amounts, and refunds are stored strictly as integers in **paise** (e.g., ₹150.00 = `15000`). Floating-point types are strictly forbidden.
  • Universal Soft Deletes: Records are never deleted via SQL `DELETE`. Entities use `is_active = false` or explicit status enums (`lost`, `revoked`) to maintain complete audit traceability.
  • Immutable Ledger: The `transactions` table is append-only. Financial adjustments or refunds are inserted as new credit/debit rows, never modified in place.

12.3 Key Relationships & Indexing Strategy

  • idx_cards_uid_hash on (card_uid, card_token_hash) — Critical composite index guaranteeing sub-5ms identity verification lookups during POS edge scans.
  • idx_txn_idempotency on (txn_id) — Absolute unique constraint preventing double-billing if an edge server retries uploading a `sync_batch`.
  • idx_txn_edge_reconciliation on (source, status, created_at) — Enables instant admin dashboard querying for offline anomaly detection and velocity audits.

Glossary

Term Definition
Identity Token An NFC card design pattern where the chip stores only a read-only cryptographic hash. Balance math is offloaded entirely to database servers, preventing card corruption.
Local Edge Server A venue laptop or mini-PC running dockerized PostgreSQL/FastAPI over local food court Wi-Fi. Guarantees sub-10ms POS checkout speeds even during complete cellular cloud blackouts.
Torn Write Hardware corruption caused when an NFC tag is pulled away from a reader while memory sectors are actively being modified.
Visual Handshake The mandatory full-screen confirmation on FizMaa POS showing exact deduction amounts before NFC scanning is activated.
